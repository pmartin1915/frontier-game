# Audit Findings

**Date:** 2026-07-18 (UTC)

## Overview
This audit reviews the "frontier" project's automated audit system, focusing on persistent issues in the `mistral-large-latest` model's output and the triage process. The analysis reveals critical flaws in the audit model's contextual understanding, leading to self-referential meta-issues, misattributions, and stale findings. Additionally, ambiguities in triage report generation reduce clarity for human reviewers. These findings underscore systemic risks to the audit pipeline's reliability and efficiency.

---

## Findings

### 1. CRITICAL: Persistent Self-Referential Meta-Issues and Misattribution
**Category:** Bug / Process Reliability
**Severity:** Critical

#### Description
The `mistral-large-latest` model continues to generate "CRITICAL" findings that:
1. **Misinterpret the audit process itself** as a code defect (e.g., claiming a "100% false-positive rate for meta-issues").
2. **Misattribute issues** to `src/engine/director.ts`, a deterministic narrative-generation module with no audit-related functionality.
3. **Hallucinate non-existent defects**, such as citing `*` or "no actual code defect exists" as issues.

Triage reports consistently label these as `false-positive`, confirming the absence of defects in `director.ts`. This pattern indicates a fundamental failure in the model's ability to:
- Distinguish between code defects and meta-comments about the audit system.
- Recognize the roles of core project components (e.g., `director.ts` is not part of the audit/triage system).

#### Impact
- **Wasted Resources:** Human and automated triage effort spent on irrelevant findings.
- **Erosion of Trust:** Undermines confidence in the audit system's ability to identify genuine defects.
- **Noise in Reports:** Obscures real issues, reducing the audit pipeline's effectiveness.

#### Evidence
- **Triage Reports:**
  - `triage-frontier-audit-20260717111413.md` (Fingerprint: `23256f2b49d3…`): "No actual code defect exists" labeled `false-positive`.
  - `triage-frontier-audit-20260717173346.md` (Fingerprint: `46b8ba2bd362…`): Cites `*` as an issue, labeled `false-positive`.
  - `triage-frontier-audit-20260717193243.md` (Fingerprint: `46b8ba2bd362…`): Duplicate of above, same rationale.
- **Prior Audit Reports:**
  - `audit-findings-1784277244632.md` and `audit-findings-1784313203180.md` document identical patterns.

#### Recommendations
- **R1.1:** Retrain or fine-tune the audit model to:
  - Explicitly exclude meta-comments about the audit process from findings.
  - Recognize the roles of core files (e.g., `director.ts` is for narrative generation, not audit logic).
- **R1.2:** Implement pre-processing filters to block findings that:
  - Reference the audit system itself (e.g., "triage system," "false-positive rate").
  - Misattribute issues to non-audit components.
- **R1.3:** Add contextual prompts to the model to clarify the scope of the audit (e.g., "Focus only on code defects in the target project, not the audit system").

---

### 2. HIGH: Stale Findings for Non-Existent Files
**Category:** Bug / Data Quality
**Severity:** High

#### Description
The audit model generates findings referencing files or paths that no longer exist in the codebase. Examples include:
- Findings labeled `false-positive` or `stale` due to missing files (e.g., `triage-frontier-audit-20260716153334.md`).
- Ambiguous references (e.g., "path" in `triage-frontier-audit-20260714205315.md`).

This suggests:
- The model operates on outdated codebase snapshots.
- Insufficient validation of file existence before generating findings.

#### Impact
- **Wasted Triage Effort:** Investigating non-existent issues.
- **Reduced Efficiency:** Delays in addressing real defects.
- **Confusion:** Developers may attempt to fix non-existent files.

#### Evidence
- `triage-frontier-audit-20260716153334.md`: Finding for `path` (Fingerprint: `e1790dbcfab6…`) labeled `false-positive` with rationale: "The file referenced in the finding no longer exists."
- Prior audit reports (`audit-findings-1784277244632.md`, `audit-findings-1784313203180.md`) document similar patterns.

#### Recommendations
- **R2.1:** Synchronize the audit model with the latest codebase state before each run.
- **R2.2:** Implement pre-validation checks to verify file existence before generating findings.
- **R2.3:** Automatically mark findings as `stale` if referenced files are deleted, with clear rationales.

---

### 3. MEDIUM: Ambiguous Triage Report Generation
**Category:** Process Improvement / Code Quality
**Severity:** Medium

#### Description
Triage reports exhibit two key ambiguities:
1. **Fingerprints as Issue Descriptions:**
   - Reports list raw fingerprints (e.g., `e1790dbcfab6…`) as "Issue" descriptions instead of the original finding text.
   - Example: `triage-frontier-audit-20260716153334.md` cites `e1790dbcfab6…` without context.
2. **Misleading Rationales:**
   - Rationales for meta-issues may incorrectly imply core files (e.g., `director.ts`) are missing.
   - Example: "The file referenced in the finding no longer exists" for a meta-issue about `director.ts` (a core file unlikely to be deleted).

#### Impact
- **Reduced Clarity:** Human reviewers must manually look up fingerprints.
- **Misinterpretation:** Ambiguous rationales may confuse developers about the status of core files.

#### Evidence
- `triage-frontier-audit-20260716153334.md`: Issue listed as `- e1790dbcfab6…`.
- `triage-frontier-audit-20260717001350.md` and `20260717011240.md`: Issues listed as `- 326ec643957a…`.

#### Recommendations
- **R3.1:** Ensure triage reports display the original finding text, not just fingerprints.
- **R3.2:** Refine rationale generation to:
  - Avoid implying core files are missing unless explicitly verified.
  - Clearly distinguish between meta-issues and code defects.
- **R3.3:** Standardize labels (e.g., use `stale` for non-existent files, not `false-positive`).
