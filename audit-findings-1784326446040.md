# Audit Findings

## Automated Audit Report for "frontier" Project Audit Process

**Date:** 2026-07-17 (UTC)

### Overview

This audit examines the recent performance of the `mistral-large-latest` audit model and its triage process for the "frontier" project. The analysis is based on the latest triage reports (2026-07-17) and prior audit findings. The primary issues identified include persistent meta-issue hallucinations, misattribution of findings, and ambiguous triage rationales. These problems continue to undermine the efficiency and reliability of the automated audit pipeline.

---

### Findings

#### 1. CRITICAL: Persistent Meta-Issue Hallucination and Misattribution by Audit Model
* **Description**:
  The `mistral-large-latest` model continues to generate "CRITICAL" findings that are meta-comments about the audit process itself, rather than actual code defects. These findings are consistently misattributed to `src/engine/director.ts`, despite triage rationales clarifying that this file contains deterministic narrative logic and is not part of the audit system. The latest triage reports (2026-07-17) show this pattern has not been resolved:
  - Findings cite non-existent defects (e.g., Git commit hashes, `*`, or claims of "no actual code defect") as issues in `director.ts`.
  - Triage labels these as `false-positive` with rationales confirming the file's functionality and lack of defects.

* **Severity**: Critical
* **Impact**:
  - Wastes triage resources and obscures genuine issues.
  - Undermines trust in the audit model's output.
  - Misrepresents the role of core engine components (e.g., `director.ts`).
* **Evidence**:
  - `triage-frontier-audit-20260717001350.md`: Finding cites a Git commit hash (`326ec643957a…`) as an issue in `director.ts`.
  - `triage-frontier-audit-20260717111413.md`: Finding claims "no actual code defect exists" in `director.ts`.
  - `triage-frontier-audit-20260717173346.md`: Finding cites `*` as an issue in `director.ts`.

* **Root Cause**:
  The audit model lacks contextual understanding of:
  - The distinction between audit meta-comments and code defects.
  - The role of `director.ts` (narrative logic, not audit triage).
  - The scope of its own analysis (codebase vs. audit process).

---

#### 2. HIGH: Ambiguous or Nonsensical Issue Descriptions in Findings
* **Description**:
  The audit model generates findings with ambiguous or nonsensical "Issue" descriptions, such as:
  - Git commit hashes (`326ec643957a…`).
  - Wildcard symbols (`*`).
  - Self-referential claims ("no actual code defect exists").
  These descriptions lack actionable context and force triage teams to infer intent, increasing cognitive load.

* **Severity**: High
* **Impact**:
  - Reduces clarity and efficiency in triage.
  - Increases risk of misclassification or overlooked defects.
  - Indicates poor output formatting or model confusion.
* **Evidence**:
  - `triage-frontier-audit-20260717001350.md`: Issue = `- `326ec643957a…``.
  - `triage-frontier-audit-20260717173346.md`: Issue = `*`.

---

#### 3. MEDIUM: Inconsistent Triage Rationale for Meta-Findings
* **Description**:
  Triage rationales for meta-findings (e.g., those citing Git hashes or `*`) are overly simplistic or misleading. For example:
  - `triage-frontier-audit-20260717001350.md`: Rationale states "no defect is described or visible in the current code," but the issue itself is a Git hash, not a code defect.
  - `triage-frontier-audit-20260717173346.md`: Rationale describes `director.ts` as "well-documented" and "deterministic," but does not address why the audit model generated a nonsensical issue (`*`).

* **Severity**: Medium
* **Impact**:
  - Triage rationales do not clearly explain the root cause of false-positives (e.g., model hallucination vs. codebase changes).
  - May lead to confusion about whether the issue lies with the audit model or the code.
* **Evidence**:
  - All three triage reports from 2026-07-17 exhibit this pattern.

---

### Recommendations

#### 1. **Immediate Mitigations for Audit Model Output**
   * **R1.1**: Implement a post-processing filter to:
     - Automatically discard findings with nonsensical issue descriptions (e.g., Git hashes, `*`, or self-referential claims).
     - Flag findings that misattribute issues to `director.ts` or other core engine files without clear evidence.
   * **R1.2**: Add guardrails to the audit model's prompt to explicitly prohibit:
     - Meta-comments about the audit process.
     - Findings that do not describe concrete code defects.
   * **R1.3**: Require the model to provide:
     - A specific code snippet or line number for all findings.
     - A clear explanation of the defect (e.g., "missing input validation" vs. "no defect exists").

#### 2. **Improve Triage Clarity and Consistency**
   * **R2.1**: Standardize triage rationale templates for meta-findings:
     - For nonsensical issues (e.g., Git hashes): "This finding appears to be a model hallucination. No actionable defect is described."
     - For misattributed issues (e.g., `director.ts`): "The audit model incorrectly attributed this finding to `director.ts`, which is a deterministic narrative module. No audit-related logic exists in this file."
   * **R2.2**: Enhance triage system logic to:
     - Detect and flag ambiguous issue descriptions (e.g., single characters, hashes).
     - Suggest re-running the audit with updated context if findings are unclear.

#### 3. **Long-Term Model Improvements**
   * **R3.1**: Retrain or fine-tune the audit model to:
     - Distinguish between code defects and audit meta-comments.
     - Recognize the roles of core engine files (e.g., `director.ts` is not part of the audit system).
   * **R3.2**: Implement a feedback loop to:
     - Track false-positive patterns (e.g., misattribution to `director.ts`).
     - Use triage labels to improve model accuracy over time.

#### 4. **Codebase Synchronization**
   * **R4.1**: Ensure the audit model always operates on the latest codebase snapshot.
   * **R4.2**: Add pre-validation checks to confirm file existence before generating findings.
