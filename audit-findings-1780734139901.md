# Audit Findings

Date: 2026-06-06 (UTC)
Automated Audit Report: Frontier Project

## Executive Summary

The Frontier project demonstrates a well-structured architecture with clear separation of concerns, particularly between game logic, director, and narrator layers, as outlined in `CLAUDE.md` and `ROADMAP.md`. The recent implementation of companion skill bonuses (ROADMAP Phase 1, item 1) has been successfully integrated and verified by existing tests.

A significant bug was identified in the `health.ts` system directly related to the next roadmap item ("Condition escalation when untreated"). The `daysUntilCritical` property of `ActiveCondition` is currently not decremented or utilized, preventing conditions from escalating as intended. Several minor code quality improvements and potential enhancements were also noted.

## Findings

### 1. Bug: Condition Escalation Not Implemented (High Severity)

**File:** `src/systems/health.ts`
**Description:** The `ROADMAP.md` explicitly states, "Condition escalation when untreated. `ActiveCondition` carries `daysUntilCritical` but it never decrements. In `src/systems/health.ts`, decrement it each day; at 0, emit a "worsened" `HealthEvent` and apply a permanent debuff."
Upon review of `src/systems/health.ts`, the `daysUntilCritical` property within `ActiveCondition` is initialized (e.g., for `Dehydration`) but is never decremented or checked to trigger critical effects. The current logic for `Dehydration` only responds to `input.waterDepleted`, not to the passage of `daysUntilCritical`. This means the core mechanic of conditions worsening over time based on their internal timer is not functional.

**Impact:** Game conditions will not escalate as designed, potentially leading to a less challenging or dynamic health system than intended. Players might not experience the full consequences of untreated conditions.

**Recommendation:** Implement the logic to decrement `daysUntilCritical` for all active conditions each day. When `daysUntilCritical` reaches 0, trigger a "worsened" `HealthEvent` and apply the specified permanent debuff (e.g., a broken bone leading to a lasting movement penalty). Extend `tests/systems/health.test.ts` to cover this new functionality.

### 2. Code Quality: Extensive `sprite.scene` Guards in `CharacterSprite` (Medium Severity)

**File:** `src/phaser/sprites/CharacterSprite.ts`
**Description:** The `CharacterSprite` class frequently uses `if (this.sprite.scene)` checks (e.g., in `playState`, `scheduleNextIdleBehavior`, `playRandomIdleBehavior`). While these guards prevent errors if the sprite's scene is destroyed before a method completes, their prevalence might indicate a potential underlying issue with sprite lifecycle management or a race condition during scene shutdown.

**Impact:** Not a direct bug, but could be a symptom of complex or fragile object lifecycles. Excessive checks can also slightly reduce readability.

**Recommendation:** Review the scene and sprite destruction flow. Ensure that `destroy()` methods for `CharacterSprite` instances are reliably called and that any asynchronous operations (like `delayedCall` timers or tweens) are properly cancelled or cleaned up *before* the scene is fully destroyed. This might involve more explicit lifecycle hooks or a centralized cleanup mechanism.

### 3. Potential Improvement: Brittle Horse Entity Detection (Low Severity)

**File:** `src/phaser/sprites/CharacterSprite.ts`
**Description:** The `isHorseEntity` flag, which controls horse-specific idle behaviors, is determined by a string check: `config.key.includes('horse')`. This approach is brittle and could break if sprite keys are renamed or if other horse-like entities are introduced that don't match the string pattern.

**Impact:** Maintenance burden; potential for future bugs if naming conventions change.

**Recommendation:** Introduce a more robust way to categorize entities. This could be a `type` property within `SpriteSheetConfig` (e.g., `entityType: 'player' | 'horse' | 'companion'`) or a dedicated enum.

### 4. Potential Improvement: Agent Bridge Snapshot Drift (Low Severity)

**File:** `src/App.tsx`
**Description:** The `snapshot` object sent to the `/api/agent/state` endpoint manually selects properties from the game state (e.g., `player: { health: s.player.health, morale: s.player.morale, ... }`). This approach is prone to drift if the `PlayerState`, `HorseState`, or `Supplies` interfaces are updated, requiring manual modification of the snapshot logic.

**Impact:** Increased maintenance effort; potential for the agent bridge to send incomplete or outdated state information if not kept in sync with state changes.

**Recommendation:** Consider creating a utility function or a more declarative schema for defining the agent snapshot. This could involve iterating over a predefined list of properties or using a deep-pick utility that is less sensitive to interface changes.

### 5. Code Quality: Uncleared `_tracks` Map in `ambiance.ts` (Low Severity)

**File:** `src/audio/ambiance.ts`
**Description:** The `_tracks` `Map` stores `Howl` instances for ambient music. While `stopAmbiance` and `switchAmbianceTrack` correctly stop and fade out audio, the `Howl` instances themselves are never removed from the `_tracks` map.

**Impact:** For a game with a fixed and small number of ambient tracks, this is unlikely to cause significant memory issues. However, in a system with a large or dynamically growing number of tracks, it could lead to memory accumulation over a long play session.

**Recommendation:** If the number of ambient tracks is expected to grow significantly or be dynamic, consider implementing a mechanism to unload and remove `Howl` instances from the `_tracks` map when they are no longer needed (e.g., when a track is unlikely to be played again for a long time). For the current scope, this is a minor concern.

### 6. Code Quality: Health Delta Accumulation Clarity (Low Severity)

**File:** `src/systems/health.ts`
**Description:** `playerHealthDelta` and `horseHealthDelta` are initialized to 0 and then incrementally modified throughout the `calculateHealth` function. While functionally correct, for complex calculations, it can sometimes be harder to trace the final value.

**Impact:** Minor impact on readability for complex functions.

**Recommendation:** For improved clarity, especially as the health system grows, consider accumulating all deltas into a temporary object or array of modifiers, and then applying them in a single step at the end of the function. This makes the "net effect" more explicit.

## Conclusion

The project is in a healthy state, with good architectural principles and a clear roadmap. The successful integration of companion skill bonuses is a positive sign of progress. The primary concern is the unimplemented condition escalation logic, which is a direct blocker for the next roadmap item and impacts core gameplay. Addressing this bug and the minor code quality points will further enhance the project's robustness and maintainability.
