# Frontier Project Audit Report

## Executive Summary
The Frontier project exhibits a robust architectural foundation, adhering well to the Three-Layer Safety Gate principle and demonstrating strong separation of concerns. The codebase generally shows good test coverage for existing features. However, the audit identified several critical and high-severity issues, primarily related to incomplete core game mechanics, asset integrity, and system robustness. Addressing these will significantly enhance gameplay depth, stability, and overall project quality. Code quality and security also present areas for improvement to ensure long-term maintainability and resilience.

## Findings Summary

### Critical Severity (1)
1.  **Missing Condition Escalation Logic**: The `ActiveCondition.daysUntilCritical` field exists but is never decremented, preventing health conditions from worsening and applying permanent debuffs. This breaks a core game mechanic.

### High Severity (6)
1.  **Visual Asset Corruption**: Both `horse_draft_base.png` and `horse_riding_base.png` sprite files appear corrupted, with malformed image data despite valid PNG headers. This will result in broken visuals.
2.  **Missing Narrative Thread Escalation**: The Director system lacks implementation for advancing narrative thread statuses based on game events, hindering dynamic storytelling.
3.  **Incomplete Companion Loyalty Momentum**: The companion desertion logic is simplistic; low loyalty combined with low morale does not currently increase desertion probability as intended, reducing narrative depth.
4.  **Biome/Season Hunting Yield Variation Missing**: Hunt yields are currently flat values, lacking the specified biome and seasonal variations, which impacts game realism and strategic depth.
5.  **River Crossing Terrain Lock Incomplete**: The `JourneyState` lacks a `riverCrossingInProgress` flag, preventing the movement system from blocking progress during river crossing encounters.
6.  **Inconsistent RNG Injection**: The `rng` parameter is inconsistently handled across various calculators (e.g., `movement.ts`, `supplies.ts`), being optional in some but required in others. This complicates testing and standardization.

### Medium Severity (8)
1.  **Missing Audio Error Boundary**: The `src/audio/ambiance.ts` module lacks a robust error boundary for Howler.js initialization failures, potentially leading to game crashes if the audio context fails.
2.  **Audio Memory Leak**: The `_tracks` Map in `src/audio/ambiance.ts` never cleans up unused Howl instances, leading to potential memory bloat during extended play sessions.
3.  **Encounter Outcome Randomization Missing**: Numeric outcomes in encounters currently use fixed values instead of varying by ±10% as specified, reducing replayability and realism.
4.  **Companion-Specific Encounter Choices Missing**: The encounter choice system lacks a mechanism to define choices that require a specific companion (`companion: CompanionId`), limiting narrative interactions.
5.  **Settlement Barter System Missing**: There is no "negotiate" choice implemented for settlement resupply encounters, reducing player agency and interaction depth.
6.  **Pending Consequence Evaluation Incomplete**: The Director system does not parse `PendingConsequence.trigger` strings against the game state, preventing dynamic narrative consequences based on game conditions.
7.  **Hardcoded Warning Thresholds**: Supply warning thresholds (`WARNING_LOW`, `WARNING_CRITICAL`) are hardcoded in `src/systems/supplies.ts`, limiting UI customization and configuration.
8.  **Redundant State Snapshots**: The agent bridge in `src/App.tsx` pushes full game state snapshots on every phase change, causing unnecessary serialization and potential performance overhead.

### Low Severity (4)
1.  **Visual Audit Manifest Outdated**: The visual audit manifest timestamp (2026-03-31) is outdated, indicating that screenshots may not reflect the current UI state.
2.  **Missing UI Elements**: Several planned UI elements are missing, including an encounter outcome summary panel, companion status display on the map, and waypoint lore tooltips.
3.  **Audio System Limitations**: The sound effects system is marked as Work-In-Progress (WIP), and the voiceover system is not yet implemented, limiting audio immersion.
4.  **Mobile Layout Responsiveness**: There is no clear documentation of mobile-specific UI considerations, and potential layout issues on smaller screens are not explicitly addressed.

### Code Quality Issues (6)
1.  **Magic Numbers in Health System**: Several hardcoded numerical values (e.g., `DEHYDRATION_HEALTH_PENALTY = -5`) are present in `src/systems/health.ts` without being extracted to constants.
2.  **Complex Conditional Logic**: Health recovery calculations and parts of the movement system contain deeply nested conditional logic, making them difficult to read, understand, and maintain.
3.  **Inconsistent Rounding**: Different systems employ varying rounding approaches (e.g., `Math.round(x * 10) / 10` vs. no rounding), leading to potential floating-point precision issues and inconsistencies.
4.  **Missing Error Handling**: Several sections, particularly in agent bridge fetch calls and audio system, catch errors but have empty or insufficient error handling blocks.
5.  **Type Safety Issues**: The codebase contains type assertions (e.g., `as unknown as Phaser.Tweens.Tween`) and uses loose typing in agent bridge command handling, which could lead to runtime type errors.
6.  **Documentation Gaps**: Several complex functions and parameters across various system files lack complete JSDoc documentation, hindering code understanding for new developers.

### Security Issues (2)
1.  **Agent Bridge Security**: In development mode, the agent bridge lacks authentication and authorization mechanisms for commands, allowing game state modification without validation. This poses a remote code execution risk if the bridge is inadvertently exposed.
2.  **Missing Input Validation**: External inputs, particularly agent bridge commands, are not consistently validated or type-checked, making the system vulnerable to unexpected or malicious input.

## Recommendations

### Critical Priority
1.  **Implement Condition Escalation Logic**:
    *   Add daily decrement of `daysUntilCritical` within the health system.
    *   Implement a "worsened" `HealthEvent` emission when `daysUntilCritical` reaches 0.
    *   Develop and apply permanent debuffs (e.g., movement penalty for broken bones) based on critical conditions.

### High Priority
1.  **Fix Corrupted Sprite Assets**:
    *   Replace `horse_draft_base.png` and `horse_riding_base.png` with uncorrupted versions.
    *   Implement asset loading validation to detect future corruption or missing assets.
2.  **Implement Narrative Thread Escalation**:
    *   Add `NarrativeThread` status advancement logic within the Director system.
    *   Define and implement event-based triggers for thread progression.
3.  **Enhance Companion Desertion Logic**:
    *   Implement a loyalty momentum calculation that considers both loyalty and morale.
    *   Integrate this momentum into the desertion probability scaling.
4.  **Implement Biome/Season Hunting Variation**:
    *   Create a `HUNTING_YIELD_BY_BIOME` data table.
    *   Implement a date-based season factor calculation to modify yields.
    *   Add ±20% randomization to hunting outcomes.
5.  **Complete River Crossing Implementation**:
    *   Add a `riverCrossingInProgress: boolean` flag to `JourneyState`.
    *   Modify the movement system to return 0 distance when this flag is active.
    *   Ensure the flag is cleared upon encounter resolution.
6.  **Standardize RNG Injection**:
    *   Refactor all calculators to consistently accept an optional `rng` parameter.
    *   Provide a `Math.random` fallback when `rng` is not provided, simplifying testing.

### Medium Priority
1.  **Add Audio Error Boundary**:
    *   Wrap Howler.js initialization in `src/audio/ambiance.ts` with a try-catch block.
    *   Implement graceful degradation or fallback to silent operation if audio context fails.
2.  **Implement Audio Memory Cleanup**:
    *   Add `unloadUnusedTracks()` logic to `src/audio/ambiance.ts` with a suitable Time-To-Live (e.g., 5 minutes).
    *   Call this cleanup function periodically or during `stopAmbiance()`.
3.  **Implement Encounter Outcome Randomization**:
    *   Add ±10% variation to all numeric encounter outcomes.
    *   Ensure actual randomized values are logged in `EventRecord` for debugging.
4.  **Implement Companion-Specific Encounter Choices**:
    *   Add a `companion: CompanionId` requirement type to encounter choice definitions.
    *   Implement choice availability filtering based on the presence and status of companions.
5.  **Implement Settlement Barter System**:
    *   Add a "negotiate" choice to settlement resupply encounters.
    *   Implement barter roll mechanics and associated outcomes.
6.  **Complete Pending Consequence Evaluation**:
    *   Implement robust trigger string parsing (e.g., `loyalty < 20`) within the Director.
    *   Add game state evaluation logic to determine when consequences should fire.
7.  **Externalize Hardcoded Warning Thresholds**:
    *   Move `WARNING_LOW` and `WARNING_CRITICAL` to `src/types/game-state.ts` as `SUPPLY_WARNING_THRESHOLDS` or similar constants.
8.  **Optimize State Snapshots**:
    *   Implement shallow comparison or selective serialization for agent bridge state snapshots.
    *   Only send changed slices of the state to `/api/agent/state` to reduce overhead.

### Low Priority
1.  **Update Visual Audit Manifest**:
    *   Capture current screenshots of all UI states.
    *   Update the manifest timestamp to reflect the current date.
2.  **Implement Missing UI Elements**:
    *   Develop and integrate the encounter outcome summary panel.
    *   Add a companion status display on the map interface.
    *   Implement waypoint lore tooltips.
3.  **Complete Audio System**:
    *   Finalize the sound effects system (currently WIP).
    *   Implement the voiceover system as outlined in `src/audio/README.md`.
4.  **Improve Mobile Responsiveness**:
    *   Document mobile-specific UI considerations and design patterns.
    *   Conduct mobile layout tests and verify touch target sizes.

### Code Quality Improvements
1.  **Extract Magic Numbers to Constants**:
    *   Identify all hardcoded numbers in `src/systems/health.ts` and other files.
    *   Extract them into clearly named, documented constants, grouped logically.
2.  **Simplify Complex Conditional Logic**:
    *   Refactor nested conditional statements into smaller, more manageable functions.
    *   Add clear comments to explain the logic flow of complex calculations.
3.  **Standardize Rounding Approach**:
    *   Define and document a consistent rounding strategy for the entire project.
    *   Implement a utility function for rounding if necessary and apply it uniformly.
4.  **Improve Error Handling**:
    *   Replace empty catch blocks with proper error logging (e.g., `console.error`, a logging service).
    *   Implement graceful degradation or error recovery paths where appropriate.
5.  **Enhance Type Safety**:
    *   Replace type assertions with proper type guards and runtime validation.
    *   Strengthen type checking for agent bridge commands and other external inputs.
6.  **Complete Documentation**:
    *   Add missing JSDoc comments for all complex functions, classes, and interfaces.
    *   Ensure all parameters, return values, and examples are thoroughly documented.

### Security Improvements
1.  **Secure Agent Bridge**:
    *   Implement authentication and authorization for agent commands (e.g., API keys, session tokens).
    *   Add command validation to ensure only expected actions and values are processed.
    *   Strictly restrict the agent bridge to the development environment only.
    *   Consider rate limiting for agent commands to prevent abuse.
2.  **Add Input Validation**:
    *   Implement comprehensive validation for all external inputs, especially agent bridge commands.
    *   Perform type checking and sanitization where appropriate to prevent injection attacks or unexpected behavior.

## Conclusion

The Frontier project is commendably structured with a strong foundation in its architectural patterns and adherence to the Three-Layer Safety Gate. The identified issues, while significant in some areas, are largely related to feature completeness and system robustness rather than fundamental design flaws. Prioritizing the critical and high-severity findings will dramatically improve the game's core mechanics, visual integrity, and narrative depth. Concurrently addressing code quality and security will ensure the project's long-term maintainability, stability, and resilience. The project is well-positioned for successful completion of its roadmap with these improvements.
