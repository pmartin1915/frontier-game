# Frontier Project Audit Report

## Executive Summary
The Frontier project maintains a strong architectural foundation with effective separation of concerns and adherence to the Three-Layer Safety Gate principle. This audit consolidates findings from previous reports and provides updated severity assessments based on recent triage results. While several critical issues have been addressed or marked as intentional design decisions, multiple high and medium severity findings remain unresolved, particularly around core game mechanics and system robustness.

## Critical Severity Findings (0)
*No unresolved critical severity findings. Previous critical issue (Missing Condition Escalation Logic) was marked as intentional design decision in triage.*

## High Severity Findings (3)

### 1. Incomplete Companion Loyalty Momentum
**Location**: `src/systems/companions.ts`
**Issue**: Desertion logic remains simplistic with no implementation of loyalty momentum calculations that should increase desertion probability when both loyalty and morale are low.
**Impact**: Companion desertion mechanics lack depth and realism, reducing narrative impact.
**Recommendation**:
- Implement loyalty momentum calculation combining loyalty and morale factors
- Add desertion probability scaling based on momentum
- Extend test coverage in `tests/systems/companions.test.ts` for desertion scenarios

### 2. Biome/Season Hunting Yield Variation Missing
**Location**: `src/systems/supplies.ts`
**Issue**: Hunt yields remain flat values with no implementation of biome/season variation as specified in ROADMAP.
**Impact**: Hunting mechanics lack environmental realism and strategic depth.
**Recommendation**:
- Add `HUNTING_YIELD_BY_BIOME` data table
- Implement date-based season factor calculation
- Add ±20% randomization to yields
- Extend test coverage in `tests/systems/supplies.test.ts`

### 3. River Crossing Terrain Lock Incomplete
**Location**: `src/systems/movement.ts`, `src/types/game-state.ts`
**Issue**: No `riverCrossingInProgress` flag exists in `JourneyState` and movement system doesn't block progress during river crossings.
**Impact**: River crossing encounters don't properly block movement, breaking core game mechanics.
**Recommendation**:
- Add `riverCrossingInProgress: boolean` to `JourneyState`
- Modify movement system to return 0 distance when flag is set
- Clear flag on encounter resolution
- Add test coverage in `tests/systems/encounters.test.ts`

## Medium Severity Findings (10)

### 1. Encounter Outcome Randomization Missing
**Location**: `src/systems/encounters.ts`
**Issue**: Numeric outcomes continue to use fixed values instead of ±10% variation as specified.
**Impact**: Encounter outcomes lack variability and realism.
**Recommendation**:
- Add ±10% variation to numeric outcomes
- Log actual values in `EventRecord`
- Extend test coverage

### 2. Companion-Specific Encounter Choices Missing
**Location**: `src/types/encounters.ts`, `src/systems/encounters.ts`
**Issue**: No `companion: CompanionId` requirement type exists for encounter choices.
**Impact**: Companion-specific interactions remain impossible.
**Recommendation**:
- Add companion requirement type to encounter choices
- Implement choice availability filtering
- Add test case for "choice requiring Coe is unavailable if Coe deserted"

### 3. Settlement Barter System Missing
**Location**: `src/systems/encounters.ts`, `src/data/encounter-templates.ts`
**Issue**: No "negotiate" choice exists for settlement resupply encounters.
**Impact**: Settlement interactions lack depth and player agency.
**Recommendation**:
- Add "negotiate" choice to settlement encounters
- Implement barter roll mechanics
- Add test coverage

### 4. Pending Consequence Evaluation Incomplete
**Location**: `src/engine/director.ts`
**Issue**: Director still doesn't parse `PendingConsequence.trigger` strings against game state.
**Impact**: Narrative consequences can't be triggered based on game state.
**Recommendation**:
- Implement trigger string parsing (e.g., `loyalty < 20`)
- Add state evaluation logic
- Implement consequence firing
- Add test coverage

### 5. Inconsistent RNG Injection
**Location**: Multiple system files (`movement.ts`, `supplies.ts`)
**Issue**: `rng` parameter remains inconsistently handled across calculators.
**Impact**: Complicates testing and standardization.
**Recommendation**:
- Standardize all calculators to accept optional `rng` parameter
- Provide `Math.random` fallback when not provided
- Document RNG usage patterns

### 6. Missing Error Handling
**Location**: Agent bridge, audio system
**Issue**: Several places continue to catch errors without proper handling.
**Impact**: Potential silent failures and debugging difficulties.
**Recommendation**:
- Add proper error logging
- Implement graceful degradation
- Add error recovery paths

### 7. Type Safety Issues
**Location**: Agent bridge, audio system
**Issue**: Type assertions remain that could fail at runtime.
**Impact**: Potential runtime errors from type mismatches.
**Recommendation**:
- Replace type assertions with proper type guards
- Add runtime validation for external inputs
- Implement type checking for agent commands

### 8. Agent Bridge Security
**Location**: Agent bridge implementation
**Issue**: No authentication/authorization for agent commands in dev mode.
**Impact**: Potential remote code execution risk if bridge is exposed.
**Recommendation**:
- Add authentication for agent commands
- Implement command validation
- Restrict to development environment only
- Add rate limiting

### 9. Missing Input Validation
**Location**: Agent bridge, external inputs
**Issue**: External input continues to lack validation.
**Impact**: Vulnerability to unexpected or malicious input.
**Recommendation**:
- Validate all external inputs
- Implement type checking for agent commands
- Add sanitization where appropriate
- Add input validation middleware

### 10. Audio Memory Leak
**Location**: `src/audio/ambiance.ts`
**Issue**: `_tracks` Map never cleans up unused Howl instances.
**Impact**: Potential memory bloat during extended play sessions.
**Recommendation**:
- Implement `unloadUnusedTracks()` with TTL (e.g., 5 minutes)
- Call cleanup during `stopAmbiance()`
- Add memory management tests
